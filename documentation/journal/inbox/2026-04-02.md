### Fix drain Temporal : court-circuit après ExecutionCompleted (2026-04-04)

**Contexte** : après le fix `WorkflowTaskRunner` → `sharedEventStore` (session précédente), l'utilisateur signale
que l'erreur gRPC persiste et pose la question de la pertinence de `InMemoryEventStore` / `NullEventStore`.

**Diagnostic** :
- `NullEventStore` dans `WorkflowTaskRunner.runtime` : correct — Temporal stocke les événements du workflow.
- `InMemoryEventStore` injecté en `sharedEventStore` pour notifier le drain : correct, mais le drain appelle
  `pollTemporalMirrorTransportsIfRegistered()` sans vérification intermédiaire.
- Séquence buguée : après la tâche 2 (workflow terminé, `ExecutionCompleted` écrit), la méthode continue et
  poll `durable_temporal_activity` via **long-poll gRPC de 60–120 secondes** → le drain ne rend pas la main.
  C'est la vraie cause de l'erreur (ou du timeout apparent).

**Correction** : `DurableMessengerDrain.pollTemporalMirrorTransportsIfRegistered()` accepte maintenant
`$eventStore` et `$executionId`, et vérifie **avant chaque transport** si `ExecutionCompleted` est présent.
Si oui, retour immédiat sans poll de `durable_temporal_activity`. Les deux call-sites mis à jour.

**Résultats** : 44/44 tests, 9 skips attendus.

### Refonte complète de la complétion synchrone Temporal (2026-04-04, suite)

**Constat utilisateur** : le drain finit toujours en timeout malgré le fix précédent, car plusieurs processus
sont en jeu. L'`InMemoryEventStore` est local au processus HTTP ; les mises à jour du worker ne le traversent
pas.

**Analyse** : `DurableMessengerDrain` est architecturalement correct pour le backend in-memory (même processus).
Pour Temporal en multi-processus, le processus HTTP doit interroger Temporal directement.

**Solution implémentée** :

1. `TemporalHistoryCursor.closeEvent()` : nouvelle méthode — un seul appel gRPC
   `GetWorkflowExecutionHistory` avec `HISTORY_EVENT_FILTER_TYPE_CLOSE_EVENT`. Retourne l'event terminal
   (completed / failed / timed-out) ou `null` si le workflow est encore en cours. Aucun parcours complet
   de l'historique.

2. `WorkflowClient.pollForCompletion(executionId, refreshIntervalMs=500, maxRefreshes=120)` :
   boucle de sondage configurable sur `closeEvent()`. Par défaut : 120 tentatives × 500 ms = 60 secondes max.
   Distingue completed / failed / timed-out. Multi-processus natif.

3. `DurableSampleWorkflowRunner.waitForWorkflowCompletion()` : routage selon le backend.
   Si `WorkflowClient` injecté → `pollForCompletion()`. Sinon → `DurableMessengerDrain` (in-memory).
   `WorkflowClient` câblé via `services.yaml` avec `'@?Gplanchat\Bridge\Temporal\WorkflowClient'`.

4. `WorkflowTaskRunner` : revert du `sharedEventStore` de la session précédente (inutile / trompeur en
   multi-processus). `ExecutionRuntime` conserve `NullEventStore` (correct : Temporal est le store).

5. `DUR005` mis à jour pour documenter le pattern `pollForCompletion` et l'architecture multi-processus.

**Résultats** : 44/44 tests, 9 skips attendus.
