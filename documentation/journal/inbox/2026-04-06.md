# Journal — 2026-04-06

## Infrastructure de tests pour `gplanchat/durable`

### Contexte

Session dédiée à la conception et l'implémentation de l'outillage de test pour les utilisateurs du composant `gplanchat/durable`. La demande couvrait trois périmètres : les tests in-memory purs, les tests d'intégration Symfony avec le bundle, et les tests d'intégration avec Temporal en CI.

### Ce qui a été produit

#### 1. Package de test in-memory (`src/Durable/Testing/`)

**`ActivitySpy`** : test double contrôlable pour les activités. Trois modes : retour fixe (`returns()`), exception (`throws()`), séquence (`returnsSequence()`). Les `Throwable` peuvent être insérés directement dans la séquence — ils sont levés lors de l'appel correspondant (approche plus propre qu'une closure). Méthodes d'assertion PHPUnit intégrées : `assertCalledTimes`, `assertCalledWith`, `assertNotCalled`, etc.

Décision de conception : `ActivitySpy` est un callable ordinaire compatible avec `RegistryActivityExecutor::register()`. Pas de classe abstraite, pas d'interface : la simplicité prime.

**`WorkflowTestEnvironment`** : façade in-memory qui agrège `InMemoryEventStore`, `InMemoryActivityTransport`, `RegistryActivityExecutor`, et `InMemoryWorkflowRunner`. Factory statique `inMemory(array $handlers, int $maxRetries)`. Expose `getEventStore()` et `getActivityTransport()` pour les assertions avancées.

**`DurableTestCase`** : classe de base PHPUnit (`extends TestCase`) qui mémorise le dernier `WorkflowTestEnvironment` créé pour brancher les assertions dessus. Méthodes : `assertWorkflowCompleted`, `assertWorkflowFailed`, `assertActivityExecuted`, `assertEventStoreContains`, `countActivityExecutions`.

#### 2. Helpers Symfony (`src/DurableBundle/Testing/`)

**`DurableBundleTestTrait`** : trait pour `KernelTestCase`. Fournit `dispatchWorkflow(class, input)` (résolution automatique du type via `WorkflowDefinitionLoader`), `drainMessengerUntilSettled(executionId)` (drain Messenger in-memory avec timeout configurable), `assertWorkflowResultEquals`, `assertWorkflowFailed`, `getDataCollector`. La logique de drain est auto-contenue dans le trait (liste de transports configurable via `$durableWorkflowTransports`).

#### 3. CI Temporal (``.github/workflows/ci.yml`)

Nouveau job `temporal-integration` : démarre `database` + `temporal` via `docker compose` depuis `symfony/`, attend que le port 7233 soit joignable (boucle `nc`, 60×3s), puis exécute `phpunit --group temporal-integration,temporal-native-spike` avec `DURABLE_DSN` injecté. Teardown `docker compose down -v` systématique (`if: always()`).

#### 4. Tests d'exemple (`tests/unit/Durable/Testing/WorkflowTestingExampleTest.php`)

9 tests qui documentent l'API par l'exemple : activité simple, activités parallèles, retry avec séquence, échec non géré, enregistrement du journal en cas d'échec, séquence épuisée (répétition de la dernière valeur), utilisation standalone sans `DurableTestCase`.

Tous passent : `OK (9 tests, 25 assertions)` en 7ms.

### Difficultés / décisions

- **`ActivitySpy::returnsSequence` avec des exceptions** : l'expression `throw` en argument d'une fonction n'est pas valide PHP. Solution : stocker les `Throwable` comme valeurs dans le tableau et les détecter dans `__invoke` pour les lever. Cela donne une API plus lisible que des closures.
- **Pas de `composer.json` root dans git** : le monorepo est structuré sans fichier root tracké dans la branche `main` actuellement — le vendor/ et les tests/ existent localement. L'autoloading PSR-4 couvre automatiquement `src/Durable/Testing/` → `Gplanchat\Durable\Testing\` car le préfixe `Gplanchat\\Durable\\` est mappé sur `src/Durable/`.
- **`DurableBundleTestTrait`** : délibérément découplé de l'app de démo (`DurableMessengerDrain` de `App\Durable`). La logique de drain est réimplémentée dans le trait avec les transports configurables.

## Nettoyage architecture noyau — étapes 1 à 5 (PR #9)

### Contexte

Session dédiée à simplifier l'architecture du noyau `gplanchat/durable` et du bridge Temporal : suppression de doublons, introduction de ports hexagonaux, et purge des dépendances fantômes.

### Ce qui a été produit

#### Étape 1 — Suppression EventSerializer

`Store\EventSerializer` était un quasi-doublon de `Mapping\EventDataMapper` (même logique de sérialisation, même mapping event-type). Tous les appels migrés vers `EventDataMapper::fromDomainEvent` / `toDomainEvent`. Fichier supprimé.

#### Étape 2 — Purge roadrunner-php/roadrunner-api-dto

Aucun fichier PHP du bridge ne l'importait. Retiré du `composer.json` du bridge (conformité DUR006).

#### Étape 3 — ChildWorkflowRunnerInterface

Création du port hexagonal `Port\ChildWorkflowRunnerInterface` (deux méthodes : `defersChildStartToMessenger()`, `runChild()`). `ChildWorkflowRunner` implémente l'interface. `ExecutionContext` et `ExecutionEngine` typés sur l'interface — plus de dépendance directe sur la classe concrète.

#### Étape 4 — UuidGeneratorInterface + NativeUuidV7Generator

Création de `Uuid\UuidGeneratorInterface` (port) et `Uuid\NativeUuidV7Generator` (RFC 9562, pur PHP, aucune dépendance framework). `ExecutionId::generate()` migré. `ExecutionContext` et `ExecutionEngine` injectent le générateur (optionnel, fallback sur `NativeUuidV7Generator`). `DurableExtension` enregistre et câble le service.

#### Étape 5 — Nettoyage composer.json du noyau

6 dépendances Symfony retirées (`symfony/uid`, `symfony/contracts`, `symfony/dependency-injection`, `symfony/config`, `symfony/routing`, `symfony/console`) + `doctrine/dbal` déplacé en `suggest` (fichiers DBAL en attente d'extraction vers `gplanchat/durable-dbal-adapter`). Le noyau ne dépend plus que de `php >=8.2`, `psr/clock`, `psr/log`.

### Décisions

- **NativeUuidV7Generator en pur PHP** : plutôt qu'un wrapper `symfony/uid`, implémentation directe de RFC 9562 avec `random_bytes()`. Élimine toute dépendance framework dans le noyau.
- **doctrine/dbal en `suggest`** : les fichiers `DbalEventStore`, `DbalActivityTransport` existent encore sur disque mais sont marqués pour suppression dans le plan de migration. Les supprimer maintenant aurait étendu le scope ; la note dans `suggest` documente la direction.

## Tâches post-CI — WorkflowClientInterface, doc utilisateur tests, amendement DUR028

### Contexte

Session dédiée aux trois tâches restantes après la stabilisation de la CI (PR #9).

### PR #11 — `WorkflowClientInterface`

`WorkflowClient` est déclaré `final` pour éviter la sous-classification accidentelle d'une classe câblée sur gRPC. Le problème : aucun test unitaire ne pouvait couvrir les consommateurs (`DurableSampleWorkflowRunner`, `TemporalWorkflowResumeDispatcher`, `TemporalReadThroughEventStore`) sans `ext-grpc`.

Solution : extraire `WorkflowClientInterface` déclarant les 7 méthodes publiques. `WorkflowClient` implémente l'interface. `DurableExtension` enregistre un alias `WorkflowClientInterface → WorkflowClient` dans le DI. `services.yaml` dans symfony/ injecte l'interface optionnellement.

Effet immédiat : `DurableSampleWorkflowRunnerRoutingTest` peut maintenant couvrir le chemin Temporal via une classe anonyme implémentant `WorkflowClientInterface`. Nouveau test `testTemporalPathDelegatesToPollForCompletion` ajouté.

### PR #12 — Documentation utilisateur tests

Création de `documentation/user/testing/_index.md` avec :
- Guide `DurableTestCase` (assertions intégrées, tableau des méthodes)
- Guide `ActivitySpy` (3 modes, inspection des appels)
- Guide `WorkflowTestEnvironment` (usage standalone)
- Guide `DurableBundleTestTrait` (prérequis config Messenger, méthodes, personnalisation)
- Arbre de décision pour choisir le bon niveau de test

### ADR DUR028 — Amendement 1

L'ADR initial documentait uniquement `pollForCompletion()`. Deux corrections postérieures nécessitaient une trace formelle :
1. L'extraction de `WorkflowClientInterface` : motivation (testabilité sans gRPC), contrat de l'interface, mise à jour du DI.
2. Le fix protobuf de `update()` : `UpdateWorkflowExecutionRequest::setInput()` n'existe pas ; l'`Input` doit être enveloppé dans `Update\V1\Request` via `setRequest()`. Documenté avec snippets avant/après.

Statut de l'ADR passé à « Accepted — amended ».
